# AST 锚点 drift 阻断的误伤率：一个基于真实历史回放的 benchmark

2026-07-13 · SpexCode

## 问题

spec 驱动开发需要一个 drift 门：受治代码在 spec 之后又动了，什么时候该阻断提交、逼人回头看 spec？
判据太松，spec 悄悄失效；太紧，无关改动（重构、注释、同文件里别的功能）反复打断开发，门就会被绕过。

SpexCode 先后用过三种判据：

- **A1，一次 drift 即阻断**：受治文件在 spec 版本之后有任何 commit 就报错。最直觉，也是最紧的下界。
- **A3，欠 3 个 commit 才阻断**：0.4.0 之前的产品策略（`driftErrorThreshold`）。想用"攒多了再拦"降低打断频率。
- **B，锚点命中才阻断**：0.4.0 现行策略（[[code-anchor]]）。spec 的 `code:` 条目可以钉住一个具名代码单元
  （`path#symbol`，函数/类/方法级，AST 提取），只有 commit 的 diff 与该单元在**当时文件版本**里的行区间相交才报错。

本报告用仓库自己的完整 git 历史构建 benchmark，量化三种判据的误伤（false block）与漏报（missed contract change）。

## Benchmark 构建

### 事件

事件 = (spec 节点, 版本窗口, commit)。对每个受治节点，取其 spec.md 的内容提交序列（纯 rename 不算版本），
相邻版本之间加上末版本到 HEAD 构成窗口；窗口内每个触碰受治文件的非 merge commit 是一个 drift 事件——
在它落地的时刻，门要回答 block 还是放行。同一 (节点, commit) 去重后共 **388 个事件**，覆盖 38 个带锚节点、
约 15 个月的真实开发史。

### 锚点标注

全仓 180 个 spec 节点中 96 个的 `code:` 指向 JS 系文件。8 个标注 agent 各读 spec body 与该文件 HEAD 的
AST 单元清单，为每个节点选出承载契约的那一个单元：**64 个选出锚点，32 个判定"整文件即契约"不设锚**
（session monolith、测试套件文件、纯类型文件、多单元等权的小模块）。选锚只看 spec 与代码，不看历史——
与真实部署时用户会做的事一致。花名册冻结在 `spec-eval/bench/drift-anchors.json`。

### 判交引擎

回放不另写判定逻辑，直接调用产品代码 `spec-cli/src/anchors.ts` 的 `anchorHitCommits`：对窗口内每个
commit，用 TypeScript 编译器 AST 从**该 commit 当时的文件内容**提取单元区间，与 `--unified=0` 的 diff
hunk 判交。单元当时不存在则不命中；当时内容解析失败按保守命中。所以测的就是线上机制本身。

### 真值：双轨

**行为轨（全量、自动、有噪声）。** SpexCode 的开发仪式把代码改动和它对应的 spec 改写熔进同一个
commit。于是每个事件天然带一个作者当时的判断：commit 同时改了该节点 spec.md（融合提交）= 作者认为动了
契约；只改代码 = 作者认为没动。388 个事件里 164 个融合、224 个纯代码。

**语义轨（抽样、盲评、主轨）。** 行为标签的噪声方向明确：懒得改 spec 的契约变更被标成阴性，顺手润色
spec 的无关改动被标成阳性。所以按 (行为标签 × 锚点判定) 四个格子分层抽 156 个事件（B 漏掉的格子全取），
每个事件由三个互不通气的评审 agent 盲判：只给"窗口起点版本的 spec body + 该 commit 对受治文件的 diff"，
问这个 diff 是否触及 spec 所述契约；三个评审各带一种视角（严格条文 / 可观察行为 / 维护者审计），多数票
为真值。评审不知道作者是否改过 spec，也不知道锚点判没判中。有效判定 148 条，其中 111 条三票一致，
37 条 2:1。真值冻结在 `spec-eval/bench/drift-truth.json`。

打分时按格子的总体规模对样本加权，还原到 388 个事件的总体。

### 行为标签有多噪

盲评结果证实了做语义轨的必要：融合提交里只有 72.1% 被判为真触及契约（其余是 spec 顺手润色、改动落在
锚外契约无关处），纯代码提交里只有 48.8% 被判为真不相关（一半其实动了契约但作者没更 spec）。两轨一致率
59.5%。行为标签适合看趋势，不适合当真值——这也是为什么主表用语义轨。

## 结果

### 主表（LLM 盲评真值，分层加权到 388 事件总体）

| 策略 | 阻断比例 | precision | recall | 误伤数（总误伤基数 135） |
|---|---|---|---|---|
| A1 一次 drift 即阻断 | 100% | 65.3% | 100% | 135 |
| A3 欠 3 commit 才阻断（旧策略） | 27.7% | 63.9% | 27.1% | 39 |
| **B 锚点命中才阻断（现行）** | **69.3%** | **75.5%** | **80.2%** | **66** |

读法：

- **A1 的三分之一强的阻断是误伤**（precision 65.3%）。这是"每次 drift 都打断"的真实代价：135 次对
  重构、注释、同文件无关改动的强制打断。
- **A3 不是 A1 的改良，是另一种失败。** 它的 precision（63.9%）不比 A1 好——commit 计数不携带任何
  语义信息，攒到 3 个再拦，拦下的东西并没有变得更相关，只是把打断延后了；代价是漏掉 73% 的真契约
  变更。它减少误伤的方式本质上是"几乎什么都不拦"。这解释了为什么该策略被退役。
- **B 把误伤砍掉一半（135 → 66，-51%），打断总量降 31%，同时保住 80% 的召回。** precision 从 65% 提到
  75%。判据从"文件动没动"细化到"契约单元动没动"，换来的是每一次打断的可信度。

### 行为轨对照（全量 388 事件，无抽样）

| | A1 | A3 | B |
|---|---|---|---|
| 纯代码事件被阻断（n=224，越低越少打断） | 100% | 34.8% | 61.2% |
| 融合事件被拦截（n=164，越高越不漏） | 100% | 15.2% | 80.5% |

趋势与主表一致：B 在两个方向上同时优于 A3，误伤显著低于 A1。

### 格子明细：B 的误伤与漏报分别是什么

| 格子（行为标签, 锚点判定） | 总体规模 | 盲评判"契约相关"比例 |
|---|---|---|
| 锚点命中、作者没改 spec | 137 | 65.0% |
| 锚点没中、作者没改 spec | 87 | 37.5% |
| 锚点命中、作者改了 spec | 132 | 86.5% |
| 锚点没中、作者改了 spec | 32 | 54.8% |

两个值得注意的事实：

1. **B 的"多拦"大半是正当的。** 锚点命中而作者没改 spec 的 137 个事件里，65% 被盲评判为真触及契约——
   即这一格主要不是 B 的误伤，而是作者的 spec 欠账。锚点门在这里起的正是设计意图中的作用。
2. **B 的漏报集中暴露单锚的上限。** 作者改了 spec 而锚点没中的 32 个事件里 54.8% 为真——契约变更落在
   锚定单元之外（一个文件承载多个契约单元、变更发生在别的函数、或 spec 语义性改写不对应锚内行）。
   这是"每节点至多一个锚"的结构性代价，也指向后续方向：多锚、或对高密度文件保持无锚（无锚节点永远
   只是 advisory，不会漏报成静默）。

### recall 缺口的解剖：漏报漏在哪，多锚能补多少

B 的 80.2% recall 是主表里最值得追问的数字。对全部 32 个已判"契约相关但锚点未命中"的事件逐个
提取当时的 AST 单元并与 diff 判交，漏报的构成是：

| 漏报类型 | 数量 |
|---|---|
| 变更落在同文件的**另一个具名单元** | 29 |
| 变更只落在模块顶层代码（不属任何单元） | 1 |
| 锚定单元在该 commit 时还不存在 | 1 |
| 该版本文件无法解析（文件被删除的 commit） | 1 |

29/32 的漏报是"可锚而未锚"：契约由同文件的多个单元共同承载（`side-nav` 锚在 `SideBar` 而变更在
`RailButton`/`ENTRIES`，`mentions` 锚在 `dispatchMentions` 而变更在 prompt 文案常量），单锚的选择
丢掉了其余承载者。这不是 AST 判交机制的失误，是"每节点至多一个锚"的标注约束太窄。

两个**自动**放宽的变体在同一真值上模拟（命中 = 单锚命中，或 diff 与变体的单元集相交）：

| 变体 | 阻断比例 | precision | recall | 误伤数 |
|---|---|---|---|---|
| B 单锚（现行） | 69.3% | 75.5% | 80.2% | 66 |
| B2 + spec 正文提到的所有单元 | 79.6% | 69.9% | 85.2% | 93 |
| B3 + 任意具名单元 | 93.7% | 69.1% | 99.1% | 112 |

两个自动变体的交换都不划算：B2 只换来 5 个百分点的 recall，误伤回升 41%（spec 正文顺带提及的
单元远多于真正承载契约的单元）；B3 的 recall 接近满，但误伤 112 已经接近 A1 的 135——"文件里任何
具名单元动了都拦"与"文件动了就拦"只差一层皮。

指向的设计结论：**补 recall 的路径是人工精选的多锚，不是自动放宽**。`code:` 允许一个条目带 2-3 个
`#symbol`（由标注者按 spec body 点名真正的契约承载单元），29/32 的漏报类型都在这条路径的射程内，
而精选避免了 B2 的发散。这是 [[code-anchor]] 的下一步候选，落地前应回到本 benchmark 重打分。

### 把观察范围扩到全部受治节点：无锚该不该恒阻断（B′）

主表只覆盖能选出单锚的 64 个节点。32 个"整文件即契约"的无锚节点贡献了另外 429 个事件——占全部
817 个事件的 53%，且集中在最高流量的 monolith（spec-cli 99 个、sessions-core 84、harness-adapter
51、guide 44）。现行产品对无锚节点**永不阻断**（advisory only），这是一个刻意的设计选择；它的对立面
是 B′：有锚按命中判，无锚恒阻断。为了给这两个选择定价，对无锚事件补了第二轮盲评（79 个事件按融合/
纯代码两格分层，同一套三视角设置，66/79 全票一致），并入冻结真值后按六格加权还原到全总体：

| 策略（全 96 节点总体，817 事件） | 阻断比例 | precision | recall | 误伤数（基数 314） |
|---|---|---|---|---|
| A1 一次 drift 即阻断 | 100% | 61.6% | 100% | 314 |
| A3 欠 3 commit（已退役） | 37.7% | 57.1% | 35.0% | 132 |
| B 锚点命中，无锚不阻断（现行） | 32.9% | **75.5%** | 40.4% | **66** |
| B′ 锚点命中，无锚恒阻断 | 85.4% | 64.9% | **90.0%** | 245 |

三个事实：

1. **全局视角戳破了主表 recall 的条件性。** B 在带锚子集内 recall 80.2%，但全总体只有 40.4%——
   无锚节点上的每个真契约变更它都放行。"无锚=advisory"等于把过半事件放在门外。
2. **B′ 不是免费的对称修复。** 无锚节点上 B′ 就是 A1，而无锚事件里纯代码提交占 71%（305/429，其中
   仅 47.5% 被判契约相关），所以 B′ 的 precision（64.9%）几乎跌回 A1（61.6%），误伤从 66 涨到 245。
   它买到的是全局 recall 90%。
3. **B 与 B′ 是同一谱系的两端，门的整体质量由锚点覆盖率决定。** 每把一个高流量 monolith 从"无锚"
   搬进"带锚"，它的事件就从 ~58% precision 的恒阻断区搬进 75% precision 的锚点区。而 monolith 之所
   以无锚，正是"没有单一主单元"——多锚（上一节的结论）同时是这两个缺口的解法。在多锚落地前，B′
   可以作为按仓库口味的配置项（要全局召回的团队选它，代价是误伤回到 A1 的八成）。

### 评审可靠性

两轮共 227 个事件。第一轮（带锚，148）三票全一致 111/148（75%）；各视角阳性率：严格条文 51.0%、
可观察行为 70.1%、维护者审计 58.7%——视角间的松紧差是设计好的多样性，多数票把它折中掉。第二轮
（无锚，79）全票一致 66/79（84%）。

### 副产品：锚点上线当天抓到 9 处现存分叉

把 64 个锚点写进 `.spec` 后跑 `spex spec lint`，9 个节点立刻报 anchor-drift 错误——全部是 main 上
现存的、未处置的契约级 spec-code 分叉（不是误报；例如 `session-attach` 的 spec 承诺错误提示里的
`reopen`/`capture` 动词，代码已改成 `resume`/`show --capture`，盲评三票均判契约相关）。这 9 个节点的
锚点本次**暂不落地**（落了会把 main 的 lint 打红），在花名册里标 `pending: true`，处置（重写 spec 或
确认后 ack）留给各节点 owner：

graph-cache · session-attach · source-of-truth · drift-by-ancestry · evals-view · evals-feed ·
session-console · session-multi-select · eval-tab

对报告主题而言这是一个现实注脚：计数门下这些分叉已经安静地存在了一段时间；锚点门装上的第一天就把
它们点了名。

## 效度威胁

- **LLM 评审当真值。** 缓解：三视角独立盲评 + 多数票，75% 全票一致；评审对两个策略信号（行为标签、
  锚点判定）双盲，不会向任一策略偏置。未做人工复核抽查，结论的绝对值应带此保留读。
- **锚点是今天选的，历史是过去发生的。** 选锚基于当前 spec body 与 HEAD 代码，可能带轻微后见之明。
  方向上这模拟的是真实采用场景（用户在"现在"装锚，门作用于"未来"），且判交时单元区间取自各 commit
  当时的文件版本，锚在历史上不存在时不命中，不会虚增召回。
- **基线很高。** 本仓的 spec 粒度贴代码（65% 的 drift 事件被判契约相关），precision 的提升空间被压缩；
  在 spec 稀疏、无关改动占比更高的仓库，B 对 A1 的 precision 差距只会更大，本仓给出的是偏保守的估计。
- **覆盖面。** 主表覆盖能选出单锚的 64 节点；32 个无锚节点由第二轮盲评与 B′ 一节单独覆盖（全总体表）。
  文件级 rename 会截断 rev-list 历史（spec 侧用了 --follow，代码侧未跟 rename），少量早期事件因此缺席。
- **A3 的近似。** 旧策略按节点累计计数报错，本文以"窗口内第 3 个及以后的事件视为被拦"近似其逐事件
  行为，方向与量级忠实，边界细节（跨窗口累计、ack 清零）未逐一复刻。

## 复现

```
npx tsx spec-eval/bench/drift-replay.ts
```

三件套：`spec-eval/bench/drift-replay.ts`（回放 + 打分，直接 import 产品判交引擎）、
`drift-anchors.json`（96 节点锚点花名册）、`drift-truth.json`（227 条冻结盲评真值，两轮 + 分层权重）。
判据改动的义务见 [[drift-replay-bench]] 节点。
