§1 目标与那个真正的难点

目标:SpexCode 是装给用户、用来开发别的项目的工具(不是为了开发它自己)。用户装好后:spex init 初始化自己的项目 → 跑 spex serve 起后端(board/dashboard/session 的根)→ 然后自己启动 claude / codex 在项目里干活,并希望 SpexCode 的整套(hook + 契约)对这些自启的 agent 生效。

难点(核心):用户自己敲 claude/codex 时,启动命令里没有任何 SpexCode 进程去拼 --append-system-prompt。所以 --append-system-prompt 结构上不可能把契约送达这些 agent。
→ 因此 SpexCode 的一切只能靠 harness 自动发现的文件到达:
  • hook → .config(surface:hook 节点)+ 裸启动 shim .claude/settings.json / .codex/hooks.json → 派发器
  • 系统提示/契约 → CLAUDE.md / AGENTS.md(唯一通道)

§16 理想路径 npm i / spex init / claude·codex(零操作) — 实测一处物理冲突

目标: npm install spexcode → spex init → 用户自启 claude/codex, 之后零人为操作, 不污染全局 claude/codex, 不覆盖原有 CLAUDE.md/AGENTS.md.

不覆盖 CLAUDE.md/AGENTS.md: ✅ managed <spexcode> 块, 只动标记间, 用户内容保留. shim/hooks/契约不进全局: ✅ .claude/settings.json·.codex/hooks.json·AGENTS.md/CLAUDE.md 全项目本地.

实测冲突: "零操作"与"不污染全局"对 Codex 不可兼得. 把 trust 写进项目本地 .codex/config.toml → codex 仍弹目录信任窗("信任目录后才加载 project-local config/hooks")→ 声明 trust 的项目配置要等信任后才被读 = 鸡生蛋, 项目本地 trust 结构上不可能(安全设计: 仓库不能自己声明可信). 故 Codex 目录+hook trust 只能写全局 ~/.codex/config.toml.
→ 要零操作: init 往全局写 [projects."<proj>"]trust_level + 每 hook 一条 [hooks.state]trusted_hash(~7 条, 全 scoped 用户自己项目路径).
→ 要零全局写: 留一次性 2 键信任弹窗.
Claude 类似但更轻(可能一次性 folder-trust 写 ~/.claude.json; 本机 0 弹窗或因 root 已信任).

取舍(唯一待定): (A) 零操作 + 全局写最小 scoped trust(只你自己项目路径几条, 不改他项目, additive) / (B) 零全局写 + 保留首次 2 键弹窗. 其余两约束都满足.

§15 等价性分析(持久 manifest + 哈希门设计)

等价定理(dashboard-Claude 路): 设 M(C)/A(C)=由 .config 内容确定性派生的 manifest/契约, C0=与今天硬编码 hook 集相等的内容(增量1已证 M(C0)==legacy). 当且仅当 "持久产物==derive(C当前)" 且 "C当前==C0" 时, 新行为=旧行为(hooks=派发器(M(C0))=legacy; sysprompt=append-system-prompt(C0)=旧, 代码没动). 关键: 今天 hook 是冻结硬编码会话内永不变; 新设计让它动态随可编辑 .config → 等价只在 ".config 没被改+无待同步陈旧" 时成立.

1. 可能不等价: (天花板)自启路系统提示层级(append=system-prompt层 vs 文件=user-message层); (天花板)Codex hook(无 file_path/Notification/StopFailure→4 个 hook no-op); (条件)派发器顺序 vs 原生并行(仅因当前 hook 独立); (新)并发会话 hook 漂移(派发器每事件重读持久 manifest, A 改 .config→并发 B 下个 PreToolUse 就换 hook, 今天冻结不会).

2. compromise: 自启系统提示降 user-message 层(harness 极限); Codex 契约 developer-context 层; 自动写 Codex trust=替用户跳过任意代码 hook 的安全审查(可接受但需告知); 陈旧窗口有界.

3. side effect: hook 现在写文件(AGENTS/CLAUDE/manifest/trust, 必须幂等+只动 spexcode 块); 写用户全局 ~/.codex/config.toml(跑出项目外, 每路径累积一条); 哈希门给每热路径 hook +~10ms, 改动时触发的 hook 吞 0.85s; 多出持久产物+marker.

4. 对不齐的瞬间: (新)会话外改动→下次会话启动读旧契约(harness 启动时读, 此前无门跑过则旧; 今天没有此窗口); (继承非回归)会话内改动改的人自己当场看不到(今天同样); (实现)重生写竞态需原子(temp+rename/锁); (实现)manifest 与 AGENTS.md 非原子配对需整组原子交换; (Codex)仅我们改 shim 模板时 trust 变 Modified 重弹, shim+trust 须一起原子写(shim 稳定故罕见).

结论: 稳态 dashboard-Claude 严格等价; 所有偏差要么是 harness 物理天花板(系统提示层级/Codex 字段), 要么是"冻结 hook→动态 .config"本身的代价(陈旧窗口/并发漂移/写副作用)=DIY 可编辑换来的, 靠原子写+哈希门压到最小. 实现里必须解决的是 §4 原子性(竞态/配对/shim+trust 同步).

§14 确定性内容哈希门(实测;取代 payload-path,后者会漏)

payload-path 检测被否(用户指正,正确): agent 用 bash/sed 改、用户自己改、别的 agent 改——payload 里都没那路径,会漏。正解 = 内容哈希门: 查文件实际内容状态, 与"谁改怎么改"无关, 一个不漏。

实测(.config+config 两根, 27 文件): 哈希门 ~10–20ms(纯 shell, 可忽略; vs node 跑 spex 一条 ~0.85s)。bash echo / sed / 新增文件 三种改法全测到 ✅; touch 不改内容→哈希不变(不误触发, 比 mtime 准); 还原→哈希复原(确定性)。

确定性算法(SessionStart 与 PostToolUse 共享同一门):
(1) H' = sha256(配置根所有 .md/.sh 内容, 排序拼接) — 纯 shell ~10ms;
(2) 与存的 H(上次物化哈希, gitignore marker)比;
(3) H'==H → 立即退出(常态);
(4) H'≠H → 才付 ~0.85s node regen: 从当前内容幂等物化 AGENTS.md/CLAUDE.md 块+shim+Codex trust, 写回 H'。只在真改时跑, 非随便全量替换。

SessionStart 不砍: manifest(派发器要读)每会话在 SessionStart 编译, 须在首个 PreToolUse 前就绪; 故 SessionStart 保留=编译 manifest+跑同一门, PostToolUse 只跑便宜门, 两者共享门逻辑非两套。(manifest 亦可用同哈希缓存: .config 没变即复用, 省 0.85s, 优化非必须。)

§13 materialization 改走 hook 层,不依赖 serve(取代 §2 的"后端物化")

调研结论: serve(index.ts)是纯 dashboard HTTP 后端(/api/board·specs·sessions·终端 socket·merge),boot 时无任何 .spec watch、无物化副作用。spec 核心(specs/lint/yatsu/hooks/dispatcher/init)全跑在 spex CLI + hook 上, 不依赖 serve; serve 只是 dashboard 数据 + 会话编排层。→ serve 不是必须一直开的; materialization 绝不该依赖它。

新方案(取代 §2 后端 watcher, 顺带消除两-watcher 混淆 + "后端没跑就缺文件"空窗):
• spex init(bootstrap): 写 shim(.codex/hooks.json·.claude/settings.json)+ 播种 .config + 写 Codex trust + 首次生成 AGENTS.md/CLAUDE.md 块。(鸡蛋: shim 必须 init 写, SessionStart hook 才触发得了。)
• SessionStart hook(全量重生·catch-all): 每次启动重生契约块(+已在做的 manifest 编译), 兜住一切非工具改动(git/批量/手改)。
• PostToolUse hook(会话内增量): agent 改 spec/.config 即更新, 给并发 agent/下次启动用。

高效探测(优化用户的三关): PostToolUse payload 自带被改路径(Claude tool_input.file_path; Codex apply_patch command)→ 第一关 = 看 payload 路径是否落在 config 根/surface 节点(纯字符串, 零 FS, 绝大多数瞬间返回); 命中→确认 surface:system→幂等替换 <spexcode> 块; SessionStart 全量兜底漏网改动. 坑: "看文件夹 mtime"——目录 mtime 改内容不变, 要 find -newer marker(比 payload 路径贵); 故 payload 路径当免费第一关 + SessionStart 兜底最优, mtime-marker 作可选加固.

§12 易误解 / 易栽跟头的点(开建前请逐条核对)

  1. 两个 watcher 别混: supervise.ts 监视 SpexCode 自身源码(自我开发); materialization 要在 index.ts 新加监视项目 .spec 的 watcher.
  2. dashboard 启动 vs 用户自启是两条不同路: dashboard 由 SpexCode 拼命令(append-system-prompt/--settings/可 bypass); 用户自启无 SpexCode 进程, 只靠自动发现文件+预写 trust. 零摩擦 trust 只服务用户自启.
  3. 源码/运行时/派生物三类别混: spec-cli/hooks 模板=源码; .config=运行时(用户可编辑, init 播种); shim/契约块/trust=派生物(后端可重生成). .config 不被 code: 治理.
  4. 系统提示层级不同: dashboard=system-prompt 层; 用户自启=user-message/developer 层(被发现文件天花板). 双重注入: 同会话 append + 文件块=注入两次, 需定规则(未拍). §9 SessionStart 兼发 与 §2 物化文件是同一目的两套机制, 别都上.

  6. 派发器暗坑: 真实阻断靠 stdout {"decision":"block"}(exit0)非 exit2, 派发器透传 stdout; 两个吐 JSON 的 hook 同事件会拼成非法 JSON. 顺序跑等价仅因 hook 互相独立. block 标志只对 exit-2 hook 有意义.
  7. manifest 时机/作用域: 每会话从自己 worktree 的 .config 编译; 分支无 .config/core hook→manifest 空→没 stop-gate! 故 sessions.ts 切换必须等 .config hook 合并到 main 后. 编辑 .config 下次 session 生效, 非运行中即时.
  8. Codex trust 写入细节: 哈希用 env 替换前原始 command, 含 timeout(600)/async; 写错→Modified 重弹. KEY 含项目路径(每 worktree 一条), 哈希值与路径无关. 写全局 ~/.codex/config.toml 必须 additive. shim 升级→哈希变→旧 trust 重弹. 安全: 替用户自动信任任意代码 hook, 如实告知.
  9. 文件落点/幂等: Codex 只认根 ./AGENTS.md(非 .codex/AGENTS.md); Claude 认 ./CLAUDE.md 或 ./.claude/CLAUDE.md. managed 块只替换标记间, 保留其余, 幂等. hideClaudeMd 与"契约入 CLAUDE.md 块"冲突, 与双重注入决定绑定.
  10. 上线顺序/后端未跑: trust 条目 per-machine 不能入库(每机 init/后端写); CLAUDE.md/AGENTS.md/shim 入库 vs 重生成(首次/clone 空窗)需定. 等价性只对 dashboard-Claude 成立, Codex 是新增, 增量3 additive 不改 dashboard 路.
  11. 已完成别重做: 增量1+2(surface:hook 机制/6 hook 节点/派发器/sessions.ts 接入/等价证据)已提交验证; 增量3 在其上加"后端物化+init 模板+Codex trust 写入".

§11 trusted_hash 算法已反推(看 codex 源码,3/3 验证)

来源 openai/codex: codex-rs/hooks/src/engine/discovery.rs::command_hook_hash + config/src/fingerprint.rs::version_for_toml。公式(已对 3 个已知样本验证通过):

trusted_hash = "sha256:" + sha256( canonical_compact_json({
  "event_name": <snake_event>,        # pre_tool_use / session_start / stop / ...
  "hooks": [{ "type":"command", "command":<cmd>,
             "timeout": <timeout 或 600>, "async": <bool> }]
}))   # canonical = 所有 key 递归排序; None 字段(matcher/commandWindows/statusMessage)经 TOML 省略

含义(回答"要不要测试/放哪"):不需要"抓一次哈希+测漂移"。后端写 .codex/hooks.json shim 时当场用此公式算出 trusted_hash 一并写入 config.toml → 哈希与 shim 同源、不可能漂。唯一值得留的测试 = 一个算法回归测试(钉死已知向量,如 pre_tool_use/echo PRE → d9c90ae8…),CI/commit 跑;若某天升级 codex 后对不上,说明 codex 改了规则、我们的计算函数要更新。细节:哈希用 env 替换前的原始 command 串(${VAR} 占位符原样参与)。

§10 我们能替用户写入信任 → 0 弹窗(实测成立)

实测:直接把信任写进 ~/.codex/config.toml,codex 启动 0 弹窗(目录信任 + hooks 审查全跳过)。两点已验证:(1) [projects."path"] trust_level="trusted" 跳过目录提示;(2) hook 的 trusted_hash 是内容哈希、与路径无关——把同一份 .codex/hooks.json 的哈希复用到新路径项目,codex 照样信任 → 对固定 shim 算一次哈希、所有项目复用即可。

[projects."<项目路径>"]
trust_level = "trusted"
[hooks.state."<路径>/.codex/hooks.json:session_start:0:0"]
trusted_hash = "sha256:<该 hook 内容的固定哈希>"   # 每事件一条

含义:spex init/后端可替用户写入这些条目 → 用户自启 codex 零摩擦(Claude 文件路本就 0 弹窗)。算法尚未精确反推(含规范序列化),但"抓一次哈希复用"已实测可行。安全:等于 SpexCode 替用户自动信任自己的已知 hook(用户装了即信任),需如实告知。

§8 Plugin 路 vs 文件路(深调研结论 + 待实测摩擦)

Claude(.claude-plugin/plugin.json)/ Codex(.codex-plugin/plugin.json)都有 plugin 系统,hook 都能打包(hooks/hooks.json+PLUGIN_ROOT)。但两者 plugin 都不能自动注入 CLAUDE.md/AGENTS.md;skill 只按需加载(渐进披露),非常在指令。仓库内 plugin 不自动生效(需 install/enable,落全局配置);无单一跨 harness plugin 包(.claude-plugin≠.codex-plugin;只 SKILL.md 可移植且需改写)——早先"跨 harness 通用 plugin"设想被证伪。

真正收获:两 harness 常在注入都只靠 SessionStart hook 的 additionalContext → sessionstart.sh 本就能兼发契约(读 surface:system → additionalContext),取消整个 CLAUDE.md/AGENTS.md 物化层(无文件物化、无 managed-block、hideClaudeMd 张力消失、动态、两 harness 通用;代价同物化天花板:user-message 层 + Claude 10k/字段分片)。

已实测 Claude 文件路 = 0 步(真实 claude TUI: 项目 .claude/settings.json 4 hook 无 trust 弹窗自动加载, /hooks 确认; CLAUDE.md 自动发现) → "hook 要逐个启用"对 Claude 不成立. Codex plugin 路 = 2 命令(marketplace add + plugin add)且 plugin 复制进全局 ~/.codex/plugins/cache/(副本). Codex 文件路: AGENTS.md 0 步(prompt-input 已证); hooks 需 trust(config.toml 逐 hook 记 hash)——逐条/批量 UX 未观测(codex 本机无 auth, 登录墙).

实测关键发现: plugin 路把 hook 复制进全局 cache → 改仓库那份不生效需重装 → 破坏 DIY 即时编辑; 文件路 hook 活在仓库 .config → 改即生效 → 合 DIY. 结论(实测支撑): 走文件路 — Claude 零摩擦、hook 仓库内可即时编辑、不进全局 cache、无单一跨 harness 包; plugin 唯一卖点(打包)盖不过这些. 两路的安装/trust 都可被 spex init 自动化(但都动全局配置).
