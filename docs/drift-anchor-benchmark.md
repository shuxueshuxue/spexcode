# AST 锚点 drift 阻断的误伤率：一个基于真实历史回放的 benchmark

2026-07-13 · SpexCode

## 问题

spec 驱动开发需要一个 drift 门：受治代码在 spec 之后又动了，什么时候该阻断提交、逼人回头看 spec？
判据太松，spec 悄悄失效；太紧，无关改动（重构、注释、同文件里别的功能）反复打断开发，门就会被绕过。

SpexCode 先后用过三种判据：

- **A1，一次 drift 即阻断**：受治文件在 spec 版本之后有任何 commit 就报错。最直觉，也是最紧的下界。
- **A3，欠 3 个 commit 才阻断**：0.4.0 之前的产品策略（`driftErrorThreshold`）。想用"攒多了再拦"降低打断频率。
- **B，锚点命中才阻断**：0.4.0 现行策略（[[code-anchor]]）。spec 的 `code:` 条目可以钉住一个具名代码单元
  （`path#symbol`，函数/类/方法级，AST 提取），只有 commit 的 diff 与该单元在**当时文件版本**里的行区间相交才报错。

本报告用仓库自己的完整 git 历史构建 benchmark，量化三种判据的误伤（false block）与漏报（missed contract change）。

## Benchmark 构建

### 目标变量：本报告的 P/R 全部测 Y1，不是 Y2

先把测的东西说死，两个变量不许混：

- **Y1（本报告全部 P/R 表所测）**——逐 commit 的定位问题：*这个* diff 是否触及 spec 所述契约。
  这是判交器/定位器的决策质量，标签属于 commit，落地即定死。
- **Y2（运行时真正关心的）**——逐状态的问题：此刻 spec 是否已失效、需要人回头看。它属于
  （节点, 时点），会随 spec 补账或 ack 而翻转。两者在 spec-先行分提交（spec 已领先、代码随后
  实现——Y1 阳性但 Y2 阴性）和 ack 之后的窗口上分道扬镳。

所以主表的 precision/recall 是**定位器口径**：给定事件，判交信号找没找对承载契约的单元。它不是
"门装上后 spec 失效少了多少"——那是行为效应，回放根本测不了：**回放只给决策质量，不主张任何因果**。
Y2 的铺垫（staleness episodes 与 R1 曝露下界）见后文专节，尚未成为计分轨。

### 事件

事件 = (spec 节点, 版本窗口, commit)。对每个受治节点，取其 spec.md 的内容提交序列（纯 rename 不算版本），
相邻版本之间加上末版本到 HEAD 构成窗口；窗口内每个触碰受治文件的非 merge commit 是一个 drift 事件——
在它落地的时刻，门要回答 block 还是放行。同一 (节点, commit) 去重后共 **388 个事件**，覆盖 38 个带锚节点、
约 15 个月的真实开发史。

### 锚点标注

全仓 180 个 spec 节点中 96 个的 `code:` 指向 JS 系文件。8 个标注 agent 各读 spec body 与该文件 HEAD 的
AST 单元清单，为每个节点选出承载契约的那一个单元：**64 个选出锚点，32 个判定"整文件即契约"不设锚**
（session monolith、测试套件文件、纯类型文件、多单元等权的小模块）。选锚只看 spec 与代码，不看历史——
与真实部署时用户会做的事一致。花名册冻结在 `spec-eval/bench/drift-anchors.json`。

### 多锚点标注（盲注协议，已计分）

单锚花名册的追问：契约是否其实由不止一个命名单元承载。`spec-eval/bench/drift-multi-anchors.json`
对全部 64 个已锚节点各给出同一 base 文件里的 1–3 个命名单元（共 119 个 selector，39 节点多单元），
0 号恒为种子锚。协议按更强的盲性执行：标注只用节点当前 spec body、code 路径、产品提取器的 AST 单元
清单与单锚种子花名册——真值、盲审队列、评审票、任何漏报/策略结果段一律不接触，**花名册先提交、后看
真值**（注释提交 d01cc930，其父即花名册内 frozenAt，先于本次打分 HEAD；这一祖先序连同"恰一个内容
提交、不携带任何冻结标签工件"都是回放验收门里的机器检查，不是口头保证）；追加锚只在 body 的某条独立
承重承诺一对一落在该命名单元上时才加，绝不为提召回而加，每条附简明理由与来源 blob oid。复现门
`npx tsx spec-eval/bench/multi-anchors-check.ts`：结构（单 base 路径、唯一、禁 bare/scoped 混用）、
提取器解析（dead/ambiguous/typeOnly 非零退出）、盲性字段扫描、字节级确定性；HEAD 漂移如实上报不判死。
**状态：已按既有规矩计分为策略 Bm**，见"B-multi：人工精选多锚的计分"一节；主表以上分数仍是单锚口径。

### 判交引擎

回放不另写判定逻辑，直接调用产品代码 `spec-cli/src/anchors.ts` 的 `anchorHitCommits`：对窗口内每个
commit，用 TypeScript 编译器 AST 从**该 commit 当时的文件内容**提取单元区间，与 `--unified=0` 的 diff
hunk 判交。单元当时不存在则不命中；当时内容解析失败按保守命中。所以测的就是线上机制本身。

### 真值：双轨

**行为轨（全量、自动、有噪声）。** SpexCode 的开发仪式把代码改动和它对应的 spec 改写熔进同一个
commit。于是每个事件天然带一个作者当时的判断：commit 同时改了该节点 spec.md（融合提交）= 作者认为动了
契约；只改代码 = 作者认为没动。388 个事件里 164 个融合、224 个纯代码。

**语义轨（抽样、盲评、主轨）。** 行为标签的噪声方向明确：懒得改 spec 的契约变更被标成阴性，顺手润色
spec 的无关改动被标成阳性。所以按 (行为标签 × 锚点判定) 四个格子分层抽 156 个事件（B 漏掉的格子全取），
每个事件由三个互不通气的评审 agent 盲判：只给"窗口起点版本的 spec body + 该 commit 对受治文件的 diff"，
问这个 diff 是否触及 spec 所述契约；三个评审各带一种视角（严格条文 / 可观察行为 / 维护者审计），多数票
为真值。评审不知道作者是否改过 spec，也不知道锚点判没判中。有效判定 148 条，其中 111 条三票一致，
37 条 2:1。真值冻结在 `spec-eval/bench/drift-truth.json`。

打分时按格子的总体规模对样本加权，还原到 388 个事件的总体。

### 行为标签有多噪

盲评结果证实了做语义轨的必要：融合提交里只有 72.1% 被判为真触及契约（其余是 spec 顺手润色、改动落在
锚外契约无关处），纯代码提交里只有 48.8% 被判为真不相关（一半其实动了契约但作者没更 spec）。两轨一致率
59.5%。行为标签适合看趋势，不适合当真值——这也是为什么主表用语义轨。

## 结果

### 主表（LLM 盲评真值，分层加权到 388 事件总体）

| 策略 | 阻断比例 | precision | recall | 误伤数（总误伤基数 135） |
|---|---|---|---|---|
| A1 一次 drift 即阻断 | 100% | 65.3% | 100% | 135 |
| A3 欠 3 commit 才阻断（旧策略） | 27.7% | 63.9% | 27.1% | 39 |
| **B 锚点命中才阻断（现行）** | **69.3%** | **75.5%** | **80.2%** | **66** |

读法：

- **A1 的三分之一强的阻断是误伤**（precision 65.3%）。这是"每次 drift 都打断"的真实代价：135 次对
  重构、注释、同文件无关改动的强制打断。
- **A3 不是 A1 的改良，是另一种失败。** 它的 precision（63.9%）不比 A1 好——commit 计数不携带任何
  语义信息，攒到 3 个再拦，拦下的东西并没有变得更相关，只是把打断延后了；代价是漏掉 73% 的真契约
  变更。它减少误伤的方式本质上是"几乎什么都不拦"。这解释了为什么该策略被退役。
- **B 把误伤砍掉一半（135 → 66，-51%），打断总量降 31%，同时保住 80% 的召回。** precision 从 65% 提到
  75%。判据从"文件动没动"细化到"契约单元动没动"，换来的是每一次打断的可信度。

### 行为轨对照（全量 388 事件，无抽样）

| | A1 | A3 | B |
|---|---|---|---|
| 纯代码事件被阻断（n=224，越低越少打断） | 100% | 34.8% | 61.2% |
| 融合事件被拦截（n=164，越高越不漏） | 100% | 15.2% | 80.5% |

趋势与主表一致：B 在两个方向上同时优于 A3，误伤显著低于 A1。

### 格子明细：B 的误伤与漏报分别是什么

| 格子（行为标签, 锚点判定） | 总体规模 | 盲评判"契约相关"比例 |
|---|---|---|
| 锚点命中、作者没改 spec | 137 | 65.0% |
| 锚点没中、作者没改 spec | 87 | 37.5% |
| 锚点命中、作者改了 spec | 132 | 86.5% |
| 锚点没中、作者改了 spec | 32 | 54.8% |

两个值得注意的事实：

1. **B 的"多拦"大半是正当的。** 锚点命中而作者没改 spec 的 137 个事件里，65% 被盲评判为真触及契约——
   即这一格主要不是 B 的误伤，而是作者的 spec 欠账。锚点门在这里起的正是设计意图中的作用。
2. **B 的漏报集中暴露单锚的上限。** 作者改了 spec 而锚点没中的 32 个事件里 54.8% 为真——契约变更落在
   锚定单元之外（一个文件承载多个契约单元、变更发生在别的函数、或 spec 语义性改写不对应锚内行）。
   这是"每节点至多一个锚"的结构性代价，也指向后续方向：多锚、或对高密度文件保持无锚（无锚节点永远
   只是 advisory，不会漏报成静默）。

### recall 缺口的解剖：漏报漏在哪，多锚能补多少

B 的 80.2% recall 是主表里最值得追问的数字。对全部 32 个已判"契约相关但锚点未命中"的事件逐个
提取当时的 AST 单元并与 diff 判交，漏报的构成是：

| 漏报类型 | 数量 |
|---|---|
| 变更落在同文件的**另一个具名单元** | 29 |
| 变更只落在模块顶层代码（不属任何单元） | 1 |
| 锚定单元在该 commit 时还不存在 | 1 |
| 该版本文件无法解析（文件被删除的 commit） | 1 |

29/32 的漏报是"可锚而未锚"：契约由同文件的多个单元共同承载（`side-nav` 锚在 `SideBar` 而变更在
`RailButton`/`ENTRIES`，`mentions` 锚在 `dispatchMentions` 而变更在 prompt 文案常量），单锚的选择
丢掉了其余承载者。这不是 AST 判交机制的失误，是"每节点至多一个锚"的标注约束太窄。

两个**自动**放宽的变体在同一真值上模拟（命中 = 单锚命中，或 diff 与变体的单元集相交）：

| 变体 | 阻断比例 | precision | recall | 误伤数 |
|---|---|---|---|---|
| B 单锚（现行） | 69.3% | 75.5% | 80.2% | 66 |
| B2 + spec 正文提到的所有单元 | 79.6% | 69.9% | 85.2% | 93 |
| B3 + 任意具名单元 | 93.7% | 69.1% | 99.1% | 112 |

两个自动变体的交换都不划算：B2 只换来 5 个百分点的 recall，误伤回升 41%（spec 正文顺带提及的
单元远多于真正承载契约的单元）；B3 的 recall 接近满，但误伤 112 已经接近 A1 的 135——"文件里任何
具名单元动了都拦"与"文件动了就拦"只差一层皮。

指向的设计结论：**补 recall 的路径是人工精选的多锚，不是自动放宽**。`code:` 允许一个条目带 2-3 个
`#symbol`（由标注者按 spec body 点名真正的契约承载单元），29/32 的漏报类型都在这条路径的射程内，
而精选避免了 B2 的发散。这条路径已按盲注协议标注并计分为策略 Bm——见下节。

### B-multi（Bm）：人工精选多锚的计分

先重申口径：**下面全部是 Y1 定位器分数**（逐 commit"这个 diff 是否触及契约"的决策质量），不是 Y2
（此刻 spec 是否失效），更不是因果主张（门装上后行为怎么变要前瞻实验）；产品阈值（`code:` 是否携带
多 `#symbol`）在此只是被检验的假设，不是被本表自动裁决的结论。

Bm 的定义：事件 block 当且仅当 diff 与花名册该节点**任一** selector 的单元区间相交——逐 selector 调
同一产品引擎 `anchorHitCommits`（单元按该 commit 当时文件内容提取）后取并集；0 号 selector 即种子锚，
所以 Bm 的阻断集结构上 ⊇ B（验收门断言，B 的 FN 只可能被转化、不可能新增）；无锚节点与 B 相同永不
block。权重沿用同一份冻结分层权重，真值沿用同一份 227 条冻结盲评。

带锚域（与 B2/B3 表同口径，388 加权）：

| 策略 | 阻断比例 | precision | recall | 误伤数（基数 135） |
|---|---|---|---|---|
| B 单锚（现行） | 69.3% | 75.5% | 80.2% | 66 |
| B2 + spec 正文提到的所有单元（自动） | 79.6% | 69.9% | 85.2% | 93 |
| B3 + 任意具名单元（自动） | 93.7% | 69.1% | 99.1% | 112 |
| **Bm 盲注多锚（人工精选）** | **74.0%** | **76.7%** | **86.9%** | **67** |

全总体（96 节点、817 加权）与动作通道：

| 策略 | 阻断比例 | precision | recall | block P | warn P | warn-capture |
|---|---|---|---|---|---|---|
| B（现行） | 32.9% | 75.5% | 40.4% | 75.5% | 54.7% | 59.6% |
| Bm | 35.1% | 76.7% | 43.8% | 76.7% | 53.4% | 56.2% |

读法：

1. **precision 与 recall 双升，Y1 上没有 Pareto 交换要付。** 带锚域 recall +6.7pt（80.2→86.9）的
   同时 precision +1.2pt（75.5→76.7），误伤 66→67（+1）。事先最担心的"加锚必稀释精度"没有发生——
   这正是盲注协议"只按承重承诺加锚、绝不为召回加锚"要买的性质。
2. **人工精选对自动放宽是全面占优。** Bm 对 B2 三轴皆优（precision 76.7 vs 69.9、recall 86.9 vs
   85.2、误伤 67 vs 93）；B3 的 99.1% recall 仍最高，但其 112 误伤与 69.1% precision 是"整文件门
   换皮"。"精度来自人指认契约单元"的论断从对照推断变成了直接测量。
3. **逐条归因（回放输出里逐事件点名）。** 判定样本内 Bm 相对 B 翻转 13 条：**12 条新 TP**（真契约
   变更从 warn 升为 block——issues-view#IssueDetail ×3、side-nav#ENTRIES ×2、issues-view#NewThreadForm、
   event-detail#EvalRemarks、mentions#stripRefSigil、public-mode#resolvePublicConfig、
   session-attach#assertLocalBackend、settings#Shortcuts、spec-search#nearestTitles），**1 条新 FP**
   （evals-view#EvalMasterDetail 上的一次契约无关改动）。新 FN 结构上为零（超集断言）。剩余漏报：
   带锚域 20 条（加权 33.2——落在任何 selector 之外的语义性契约变更，dashboard-shell/mentions/
   dashboard-issues 等），无锚节点 52 条（加权 249.8）依旧按设计走 advisory，多锚不改变"无锚=不拦"
   的产品选择。
4. **全总体的抬升是温和的（recall 40.4→43.8）**，因为过半事件在无锚节点上，Bm 根本不碰它们——
   多锚补的是带锚域的单锚上限，不是无锚缺口（那是 B′ 一节的议题）。
5. **对产品的含义止步于此：** 本仓历史上，"`code:` 携带 1–3 个人工点名的 `#symbol`"这一假设在定位器
   口径下成立且无精度代价。落地是 [[code-anchor]] 的独立改动（selector 语法、lint、材料化都不在本节点
   射程内），且按本节点规矩，那次改动要自带复跑表。

Bm 分数已入冻结基线（`drift-baseline.json`，含带锚域子表与通道行）；花名册的不可变与盲源自此由验收门
机器盯守（工作树=HEAD 字节一致、历史恰一个内容提交、注释提交不携带 truth/基线/队列、frozenAt ⊑ 注释
提交 ⊏ HEAD、Bm⊇B）。

### baseline 对照：git 自带的 -L 函数跟踪

一个自然的质疑是：判交不用自己写，`git log -L :funcname:file` 本来就会跟踪函数——它够不够？在
同一批已判带锚事件上，把 B 的命中信号换成"该 commit ∈ `git log -L :锚点符号:受治文件` 的输出集"：

| 判交实现（带锚域，388 加权） | precision | recall | 误伤数 |
|---|---|---|---|
| B 单锚 AST（现行引擎） | 75.5% | 80.2% | 66 |
| git log -L | 69.6% | 76.4% | 85 |

-L 对全部 34 个被判节点的符号都解析成功（没有覆盖问题），但两个轴都更差：两者命中信号一致率
78.4%，-L 多咬的 23 个事件里只有 10 个真涉契约（regex 函数头匹配 + 跨提交模糊行跟踪把邻近改动
也算进函数），B 独有的 9 个命中里 7 个为真。结论：**按历史版本逐 commit 重新做 AST 精确提取，
相对 git 内置的启发式行跟踪是真实收益**——精度来自"人指认契约单元 + AST 精确判交"这对组合，
拆掉任何一半都变差（对照 B2/B3 拆掉"人指认"的结果）。复现脚本 `gitL-compare.mjs` 在
spexcode-reports 的 pipeline 里。

### 把观察范围扩到全部受治节点：无锚该不该恒阻断（B′）

主表只覆盖能选出单锚的 64 个节点。32 个"整文件即契约"的无锚节点贡献了另外 429 个事件——占全部
817 个事件的 53%，且集中在最高流量的 monolith（spec-cli 99 个、sessions-core 84、harness-adapter
51、guide 44）。现行产品对无锚节点**永不阻断**（advisory only），这是一个刻意的设计选择；它的对立面
是 B′：有锚按命中判，无锚恒阻断。为了给这两个选择定价，对无锚事件补了第二轮盲评（79 个事件按融合/
纯代码两格分层，同一套三视角设置，66/79 全票一致），并入冻结真值后按六格加权还原到全总体：

| 策略（全 96 节点总体，817 事件） | 阻断比例 | precision | recall | 误伤数（基数 314） |
|---|---|---|---|---|
| A1 一次 drift 即阻断 | 100% | 61.6% | 100% | 314 |
| A3 欠 3 commit（已退役） | 37.7% | 57.1% | 35.0% | 132 |
| B 锚点命中，无锚不阻断（现行） | 32.9% | **75.5%** | 40.4% | **66** |
| B′ 锚点命中，无锚恒阻断 | 85.4% | 64.9% | **90.0%** | 245 |

三个事实：

1. **全局视角戳破了主表 recall 的条件性。** B 在带锚子集内 recall 80.2%，但全总体只有 40.4%——
   无锚节点上的每个真契约变更它都放行。"无锚=advisory"等于把过半事件放在门外。
2. **B′ 不是免费的对称修复。** 无锚节点上 B′ 就是 A1，而无锚事件里纯代码提交占 71%（305/429，其中
   仅 47.5% 被判契约相关），所以 B′ 的 precision（64.9%）几乎跌回 A1（61.6%），误伤从 66 涨到 245。
   它买到的是全局 recall 90%。
3. **B 与 B′ 是同一谱系的两端，门的整体质量由锚点覆盖率决定。** 每把一个高流量 monolith 从"无锚"
   搬进"带锚"，它的事件就从 ~58% precision 的恒阻断区搬进 75% precision 的锚点区。而 monolith 之所
   以无锚，正是"没有单一主单元"——多锚（上一节的结论）同时是这两个缺口的解法。在多锚落地前，B′
   可以作为按仓库口味的配置项（要全局召回的团队选它，代价是误伤回到 A1 的八成）。

### warn 通道计分：B 的"低阻断率"由 advisory 通道买单

前面所有表只给 block 打分——但产品是两级的：没被 block 的 drift 事件仍以 advisory `drift` WARN
浮出。不给 warn 计分，就等于默许把漏报藏进没人量的通道。在同一份冻结真值上，把每个事件按策略
归入**恰好一个**动作通道（block / warn / silent，互斥无重复计数；四个策略下 silent 结构性为空——
只有连 advisory 通道也设门槛的策略才会产生 silent），得：

| 策略 | block 占比 | block P | warn 占比 | warn P | warn-capture |
|---|---|---|---|---|---|
| A1 一次即阻断 | 100% | 61.6% | 0% | n/a | 0% |
| A3 欠 3 计数 | 37.7% | 57.1% | 62.3% | 64.3% | 65.0% |
| B 锚点命中（现行） | 32.9% | 75.5% | 67.1% | 54.7% | **59.6%** |
| B′ 无锚恒阻断 | 85.4% | 64.9% | 14.6% | 42.2% | 10.0% |

warn-capture = 真契约变更中被降级到 advisory 通道的份额。读法：**B 的全局 recall 40.4% 不是
"漏成静默"，而是把 59.6% 的真契约变更交给了一条 precision 只有 54.7% 的 warn 通道**——B 与 B′
之争的真实标价是"这六成变更值不值一次强打断"，而不是 recall 数字本身。warn 通道的疲劳效应
（precision 低于阈值后通道被整体无视）在本回放里测不了，属行为效应，见效度威胁。

### staleness episodes 与 R1 曝露下界（Y2 铺垫，非计分轨）

Y2 的自然单位不是事件而是 **episode**：从第一个未入账的真契约变更（onset）到 spec 补账为止。
确定性重组规则（不猜、只认历史撑得住的）：

- **onset**：窗口内第一个"已判真 & 非融合"的事件——仅当它之前的每个事件都被判过（"第一个"
  是事实而非猜测）才算 sound；有未判前缀的窗口整段记为 unresolvable，如实上报。
- **resolution**：窗口收尾的 spec 版本提交，或 onset 之后、窗口之内命名该节点的 `Spec-OK` ack
  （按祖先关系判定，不按时间戳）。末窗口无收尾则 episode 记 open、在 HEAD 处截尾。
- **R1 曝露下界**：onset 之后、resolution 之前触碰受治文件的 commit 里，`Session:` trailer 与
  onset 所属 session 不同的**去重 session 数**。无 trailer 的 commit 记 unattributed，不计入——
  R1 是 git-only 的**下界**：commit 之外的 spec 阅读对 git 不可见，永远量不到。

当前冻结真值下的产出：175 个含已判事件的窗口里，**27 个 sound episodes**（其中 5 个由 ack 收尾、
4 个 open）、86 个全判定无 episode、**66 个 unresolvable**。已重组出的 episode 里本仓的补账很快
（dwell 多为 1–2 个 commit、两天内；R1 多为 0–1 个外部 session）——但这只是可重组子集的描述，
不是总体估计。**数据边界要如实说**：66/175 不可重组的根因是现有真值按（行为×锚点）四格分层抽样，
天然不保证窗口前缀齐全。要把 Y2 变成正式计分轨，下一轮盲评必须**按窗口抽样、按前缀补齐**，
而不是扩格子。

### 人工盲审队列：40 行，等人、不许机器代填

LLM 评审当真值是本报告最大的效度保留。为此固化了 `spec-eval/bench/human-audit-queue.json`：
从 227 条冻结真值里按六格总体占比（最大余数法）确定性抽 40 行（格内按 sha256(id) 排序取前 n，
无随机数，逐次再生 byte 相同），每行只带盲评上下文——窗口起点版本的 spec body 与该 commit 对
受治文件的 diff 的 `git show` 指令（40 行×2 条指令已验证全部可解析）——**不带** truth/votes/格子/
锚点判定/行为标签任何一项。人按行填 `humanVerdict`（true/false）与可选 `humanNote`，按 id 回join
冻结真值即得 LLM 评审的校准误差。规矩：**模型永远不许填这两个字段**；填满之前，本报告的真值
不得称作"经人工验证"。回放每次运行都校验该文件（确定性再生一致、无标签泄漏、已填行数如实打印）。

两轮共 227 个事件。第一轮（带锚，148）三票全一致 111/148（75%）；各视角阳性率：严格条文 51.0%、
可观察行为 70.1%、维护者审计 58.7%——视角间的松紧差是设计好的多样性，多数票把它折中掉。第二轮
（无锚，79）全票一致 66/79（84%）。

### 副产品：锚点上线当天抓到 9 处现存分叉

把 64 个锚点写进 `.spec` 后跑 `spex spec lint`，9 个节点立刻报 anchor-drift 错误——全部是 main 上
现存的、未处置的契约级 spec-code 分叉（不是误报；例如 `session-attach` 的 spec 承诺错误提示里的
`reopen`/`capture` 动词，代码已改成 `resume`/`show --capture`，盲评三票均判契约相关）。这 9 个节点的
锚点本次**暂不落地**（落了会把 main 的 lint 打红），在花名册里标 `pending: true`，处置（重写 spec 或
确认后 ack）留给各节点 owner：

graph-cache · session-attach · source-of-truth · drift-by-ancestry · evals-view · evals-feed ·
session-console · session-multi-select · eval-tab

对报告主题而言这是一个现实注脚：计数门下这些分叉已经安静地存在了一段时间；锚点门装上的第一天就把
它们点了名。

### 父摘要 pressure 轨：子 spec 版本压父 body（描述轨，不定级）

代码 drift 之外还有第二种失效面：**直接子节点的 spec 内容版本可能把摘要型父 body 压成过期**
（父的职责就是概括子域）。本轨把它做成与主回放同命令的确定性枚举——**只描述，不实现任何运行时
lint/gate 行为，不选 block vs warn**；定级要等 π_p（见下）与前瞻实验。

语义规定（`pressure-track.ts`，全部次序只用 git 祖先关系、永不看时间戳）：

- **事件** = 直接子 spec.md 的内容版本（纯 rename 不算）且未在同一 commit 融合更新父。融合提交
  = 摘要被同步维护，无失效间隙，计 co-versioned；父 spec 在该 commit 祖先里尚不存在的计
  parent-not-yet-born 边界（多为后建分组节点收编了带历史的子节点），如实报、不猜。
- **归结四通道**：父内容更新 / 命名该父的 `Spec-OK` ack / parallel（平行 tip 上不可比的双归结，
  如实报不仲裁）/ open。**ack 只覆盖被点名的父与 ack 祖先可见的子版本**——平行 tip 上未并入的
  子版本不被覆盖；trailer 可观察，ack 是否真经重读**不可观察**，按边界上报。
- **pressure 是谓词不是计数**：open 事件数是下次改写的批量待办，不是严重度倍数；一次父改写/ack
  成批覆盖全部祖先可见待办（batching）。父更新可向上再压它自己的父（深度严格递减→有限收敛），
  永不向下。
- **masking 副作用上报**：归结性父更新同时 bump 父自己的 spec 版本，会重置父自身 `code:` 文件的
  code drift 窗口——每个归结更新都报"它会不会掩掉父自己的待处理 drift"，只上报不裁决意图。
- **曝露下界**：未归结 span 内兄弟子版本上的外来 `Session:` trailer 去重数——git-only **下界**
  （无 trailer 记 unattributed；spec 阅读对 git 不可见）。

2026-07-13 HEAD 上的回放（这些数随 HEAD 演化，不入基线；只描述本仓的历史形态）：

- 851 个直接子版本 → **401 个 pressure 事件** + 128 co-versioned + 322 parent-not-yet-born
  （完备三分，验收门断言）。归结：**更新 206 · ack 95 · parallel 32 · open 68**。
- **batching 是真实行为**：333 个已归结事件收拢成 73 批（均值 4.6、最大 29）——父摘要的维护
  节奏天然是"攒一批、一次改写"，而不是逐子响应。这是"pressure 谓词 + 批量覆盖"语义的历史依据。
- 高压父集中在高扇出的包级节点：spec-cli（130 事件，曝露 ≥58 个外来 session）、sessions（100，
  ≥46(+48 无署名)）、spec-forge（46，≥20）。12 个父在 HEAD 处于 pressed 状态（含 spec-eval 9 个
  open——本报告自己的落地也在压它）。
- **masking 不是假想**：spec-cli 的 16 次归结更新里 14 次落在自身 `code:` 文件有待处理 drift 的
  窗口上，session-console 2/2、issues 2/5——"为收拢子版本而改写父 body"与"父自身代码 drift 被
  版本重置吞掉"在历史上大量同发。任何未来的 pressure 机制都必须把这个副作用摆到明面。
- **π_p（pressure 命中里父 body 真需重写的比例）不可由 git 得出，本轨不估**：
  `pressure-audit-queue.json` 固化 40 行按父深度分层（每非空层保底 3 行、层内 sha256(id) 序、
  抽样框冻结在生成时 HEAD——后续提交不重排样本、逐次再生 byte 相同）的盲审队列，每行只带"压点时刻的父 body + 子 spec diff"两条 `git show`
  指令（40×2 已验证全部可解析），不带归结/深度/曝露任何引擎字段。规矩同 Y1 队列：**机器永远
  不许填**；填满前，上面的 401 只是事件计数，不得读作"父真需重写 401 次"。
- 七条可执行断言进验收门：仅向上传播、深度递减有限收敛、反环（归结者不得先于压点）、batching
  恰分割、无向下振荡、分类完备、队列确定性+盲。任一破即非零退出。

## 效度威胁

- **回放不主张因果。** 全部数字是"对着标签打分的决策质量"。门装上后开发者行为怎么变（block 是否
  真的缩短失效时长、会不会养出橡皮图章 ack、warn 通道会不会疲劳失效）回放测不了，需要前瞻实验
  （节点级随机分组，按共享文件聚簇），不在本报告范围内。
- **Y1≠Y2。** 主表的 precision 在 spec-先行分提交上会把"实现已被 spec 覆盖"的 block 计为正确
  （Y1 阳性、Y2 阴性）。本仓仪式以融合提交为主、此序列占比低，但换一个 spec-先行文化的仓库，
  同一张表会高估门的运行时价值。episode 轨成熟前，读表时保留此口径差。
- **LLM 评审当真值。** 缓解：三视角独立盲评 + 多数票，75% 全票一致；评审对两个策略信号（行为标签、
  锚点判定）双盲，不会向任一策略偏置。人工复核通道已固化（40 行盲审队列，见上节）但尚未填——
  填满前，结论的绝对值应带此保留读。
- **锚点是今天选的，历史是过去发生的。** 选锚基于当前 spec body 与 HEAD 代码，可能带轻微后见之明。
  方向上这模拟的是真实采用场景（用户在"现在"装锚，门作用于"未来"），且判交时单元区间取自各 commit
  当时的文件版本，锚在历史上不存在时不命中，不会虚增召回。多锚花名册（Bm）同此；另一层保留：标注
  agent 与真值评审 agent 虽互相盲（协议 + 机器门），但同源于相近的模型谱系，对"什么算承重承诺"的
  口味可能相关——这会让 Bm 的 precision 偏乐观，人工盲审队列填毕后可校准。
- **基线很高。** 本仓的 spec 粒度贴代码（65% 的 drift 事件被判契约相关），precision 的提升空间被压缩；
  在 spec 稀疏、无关改动占比更高的仓库，B 对 A1 的 precision 差距只会更大，本仓给出的是偏保守的估计。
- **覆盖面。** 主表覆盖能选出单锚的 64 节点；32 个无锚节点由第二轮盲评与 B′ 一节单独覆盖（全总体表）。
  文件级 rename 会截断 rev-list 历史（spec 侧用了 --follow，代码侧未跟 rename），少量早期事件因此缺席。
- **A3 的近似。** 旧策略按节点累计计数报错，本文以"窗口内第 3 个及以后的事件视为被拦"近似其逐事件
  行为，方向与量级忠实，边界细节（跨窗口累计、ack 清零）未逐一复刻。
- **pressure 轨的真值缺口。** 401 只是"子版本落地且父未同步"的事件计数，父 body 是否真被压过期
  （π_p）要等盲审队列填毕，本轨拒绝自估。parent-not-yet-born 排除了 38% 的子版本（后建分组节点
  收编带历史子节点），对"分组建立之前"的摘要压力本轨无话可说；ack 是否真经重读、commit 之外的
  spec 阅读，git 一概不可观察——ack 语义与曝露都只敢取 git 可见面（下界），不做推断。
