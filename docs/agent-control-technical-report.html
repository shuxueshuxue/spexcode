Claude Code / Codex TUI 外部控制链路技术报告

本报告整理一次针对 Claude Code 和 Codex TUI 的底层控制面探索。重点不是 hook、transcript、读屏、模拟键盘， 而是运行时 inspector、内部队列、本地 rendezvous socket、Codex app-server JSON-RPC、以及 Claude Channels 插件通道。

生成时间：2026-06-14 CST。主要 transcript： ~/.codex/sessions/2026/06/13/rollout-2026-06-13T16-16-56-019ec00e-31e5-7dd2-885a-df7e7dbad818.jsonl。 AgentBridge 源码快照：/tmp/agent-bridge，HEAD 1dc5e4f53568af77c5facabaae9b04f35a1e0626。

敏感性边界。 Bun heap snapshot、进程环境、debug log、app-server thread/read 都可能包含 token、prompt、路径、账号和会话内容。 本报告只记录结构、marker 和脱敏后的事实，不保留原始 heap dump、完整 env、完整 transcript 或远端请求正文。

一页结论

Claude Code 最硬链路

CLAUDE_BG_BACKEND=daemon + CLAUDE_BG_RENDEZVOUS_SOCK 会打开 Unix socket。

外部进程写入一行 {"type":"reply","text":"..."}，会进入 TUI 内部 hw(...) command queue，不碰 PTY。

Codex 最硬链路

Codex TUI 可以连接 app-server。外部客户端通过同一个 app-server JSON-RPC 发 turn/start、turn/steer、thread/inject_items。

更隐蔽的是 plain codex 在默认 Unix socket 已存在时会自动复用本地 app-server。

AgentBridge 链路

Codex 侧用 app-server/proxy 截获 agent message；Claude 侧用 Claude Code Channels Research Preview。

最后一跳是 MCP stdio notification：notifications/claude/channel。

链路 是否 TUI 有头 是否不碰 PTY 输入 能做什么 主要限制
Claude Bun inspector 是 是 读 runtime、源码 bundle、heap snapshot、状态痕迹 必须启动时打开；Bun CDP 域不完整
Claude bg rendezvous socket 是 是 注入 user prompt、slash command、bash mode、compact 必须启动时带 bg env；只接受 reply 等有限协议
Codex explicit app-server 是 是 turn/start、turn/steer、thread/read、raw assistant inject TUI 必须用 --remote 接入该 server
Codex implicit daemon reuse 是 是 plain codex 自动挂默认 socket 后可被外部控制 前提是默认 socket 已存在；若带 -c 等覆盖会失效
AgentBridge Claude channel 是 是 把 Codex agent message push 到 Claude Code channel Claude 必须加载 AgentBridge plugin/channel

证据来源和方法

这次探索用了三类证据：本机真实进程实验、源码定位、以及 transcript 回放校对。核心原则是先打印事实，再判断； 对失败实验也明确丢弃，不把时序 bug 包装成产品结论。

  • 本机 Claude Code：/opt/homebrew/bin/claude -> /opt/homebrew/Caskroom/claude-code/2.1.150/claude。
  • Claude Code 2.1.150 是 213MB arm64 Mach-O，带 __BUN 段，运行时是 Bun/JSC/WebKit 堆。
  • 本机还存在 npm 版 @anthropic-ai/claude-code@2.0.28，但它不是当前 PATH 里的 TUI 工件。
  • 本机 Codex：实验时为 0.137.0；源码审查另拉了 openai/codex 上游快照。
  • AgentBridge：clone 到 /tmp/agent-bridge，读源码而不是只读 README。

Claude Code 工件形态

当前 Homebrew Cask 版不是普通 node cli.js。它是 Bun 打包单文件，入口被编译进 Mach-O。 这让 NODE_OPTIONS=--inspect、Node preload、CommonJS require hook 这类路基本不适用于当前真实 TUI。 

which claude
/opt/homebrew/bin/claude

ls -l /opt/homebrew/bin/claude
/opt/homebrew/bin/claude -> /opt/homebrew/Caskroom/claude-code/2.1.150/claude

vmmap 和 sample 看到 JS VM Gigacage、JS JIT generated code、WebKit Malloc、 Bun Pool、HTTP Client 线程、临时 .node 和 tokio-rt-worker。这说明 Claude Code 有 JS/TUI 层，也有 native async 层。

Claude Bun Inspector 链路

入口

--inspect 作为 CLI 参数没有打开 inspector；真正有效的是 Bun 环境变量。 关键写法是指定端口和固定 WebSocket path： 

BUN_INSPECT=9235/inspect claude --dangerously-skip-permissions

# CDP websocket:
ws://[::1]:9235/inspect

# version endpoint:
curl --noproxy '*' -g 'http://[::1]:9235/json/version'

注意两点：Bun inspector 监听 IPv6 loopback ::1；本机代理环境可能劫持 loopback HTTP，所以 curl 需要 --noproxy '*'。

实测能力

CDP / Runtime 面结果
Runtime.evaluate可用，可读 process.argv、process.cwd()、process.memoryUsage()、Object.keys(Bun) 等。
Debugger.enable可用，能收到 scriptParsed。
Debugger.getScriptSource可用，能拉出主 bundle。
HeapProfilerCDP domain 不存在。
Runtime.getHeapUsage / getIsolateId不存在。
Runtime.queryObjects不存在，不能像 V8/Chrome 那样枚举活对象。
Debugger.setBreakpointOnFunctionCall不存在，抢闭包变量困难。

主 bundle 抽取

inspector 里能看到当前运行中的 BunFS 主脚本，而不是磁盘上的 npm 版源码： 

url: /$bunfs/root/src/entrypoints/cli.js
length: 15313127 bytes

这条很重要。Cask 单文件看起来不可读，但运行时通过 Debugger.getScriptSource 能把 bundle 原文拿出来。 实验中主 bundle 被抽到 /tmp/claude-2.1.150-cli.js 做静态索引。

不用 PTY 输入获取 /status 数据

/status 不是 shell 命令，而是 Claude Code 内部 React/local-jsx slash command。 源码定位到 command object： 

{
  type: "local-jsx",
  name: "status",
  description: "Show Claude Code status...",
  immediate: true,
  load: () => ...
}

组件路径被拆成： 

Status tab = Sm3() + bm3({ mainLoopModel, mcp, theme, context }) + await sWK()

其中静态行来自类似 Sm3() 的模块状态：version、session name、session id、cwd、auth/provider。 live state 来自 app store：mainLoopModel、effort、mcp.clients。

一次 heap 侧实验拿到的数据：

mainLoopModel: GLM-X-Preview
effortValue: xhigh
mcp.clients: []
cwd: /Users/lexicalmathical/Codebase/homeworks/llm
Version: 2.1.150

这条路径不向 PTY 发送 /status，也不解析屏幕。它从源码和 heap/app state 还原 /status 会展示的数据。 限制是 Bun inspector 缺少 V8 的活对象枚举和作用域读取能力，闭包变量不能随便从 global eval 里取出来。

Claude bg rendezvous socket 注入链路

启动条件

Claude Code 内部有一个 background daemon rendezvous 通道。普通前台 claude 默认不打开； 但如果启动时带下面两个环境变量，TUI 进程会创建 Unix socket： 

CLAUDE_BG_BACKEND=daemon
CLAUDE_BG_RENDEZVOUS_SOCK=/tmp/cc.sock
claude --dangerously-skip-permissions

另一个进程写一行 JSON，以换行结束：

{"type":"reply","text":"hello from socket"}

内部链路是：

Unix socket line
  -> bg rendezvous parser
  -> kb3(...)
  -> if type === "reply"
  -> hw({ mode: YR(text), value: Qy(text), priority: "next" })
  -> Claude TUI command queue
  -> normal slash parser / prompt runner / bash mode

协议语义

源码和实验显示它至少接受这些消息类型：

  • { "type": "reply", "text": "..." }：把用户回复塞进内部队列。
  • { "type": "repaint" }：请求 repaint。
  • { "type": "shutdown" }：管理器关闭后台进程。
  • { "type": "attacher-caps", "caps": ... }：外部 attacher 更新能力。

这是后台 agent / daemon 管理器和 Claude Code 子进程之间的 rendezvous，不是隐藏后门。正常用途是让后台任务没有稳定交互终端时仍能接收用户回复、重绘、关停和上报状态。

实验 1：注入 /status

写入：

{"type":"reply","text":"/status CODEX_BG_RV_1781343378"}

debug log 证明入队：

[bg-rv] enqueued reply: /status CODEX_BG_RV_1781343378

heap 里看到真实队列对象：

{
  value: "/status CODEX_BG_RV_1781343378",
  mode: "prompt"
}

实验 2：真实 user prompt 触发 API

写入普通 prompt：

{"type":"reply","text":"CODEX_NORMAL_USER_1781343432: reply with OK only"}

heap 中出现：

{
  role: "user",
  content: "CODEX_NORMAL_USER_1781343432: reply with OK only"
}

实验中曾试图把 ANTHROPIC_BASE_URL 指向本机空端口避免真实请求，但 Claude 从配置恢复到了原 provider， debug log 出现 [API REQUEST] /api/anthropic/v1/messages source=repl_main_thread。这说明注入走了真实主循环。

实验 3：tmux 可见真实 agent

起可见 TUI：

tmux new -s cc-bg-rv-real
CLAUDE_BG_BACKEND=daemon \
CLAUDE_BG_RENDEZVOUS_SOCK=/tmp/cc-bg-rv-real.sock \
claude --dangerously-skip-permissions

外部 socket 注入：

BG_RV_REAL_1781343848
BG_RV_WATCH_174450
BG_RV_STOP_174826

debug 证据：

[bg-rv] enqueued reply: ...
[API REQUEST] /api/anthropic/v1/messages source=repl_main_thread

这证明它不是只改 UI state，而是把 message 送进真实 agent loop 并触发真实 API。

实验 4：bash mode

reply.text 以 ! 开头会被 YR(text) / Qy(text) 识别为 bash mode。 实验注入： 

!printf '%s\n' 'BG_RV_BASH_175429' > '/tmp/cc-bg-rv-real.bash-proof.txt'

结果文件：

/tmp/cc-bg-rv-real.bash-proof.txt
BG_RV_BASH_175429

这说明同一非 PTY 通道不只可以发 prompt，还可以进入 Claude Code 的 bash 工具链。

实验 5：slash command 和 compact

先按用户拼写原样发 /comapct，只证明入队，不触发 compact。随后发正确的： 

/compact BG_RV_COMPACT_175922

debug 证据：

[bg-rv] enqueued reply: /compact BG_RV_COMPACT_175922
[API REQUEST] /api/anthropic/v1/messages source=compact
Forked agent [compact] received message: type=assistant
Forked agent [compact] finished: 1 messages, types=[assistant],
  totalUsage: input=1300 output=3219 cacheRead=71424 cacheCreate=0

结论：socket 注入会经过 Claude Code 自己的 slash parser；/compact 被识别为内部 compact source， 发起专门的 compact API 请求并完成 forked compact agent。

还没打穿的事

直接伪造 assistant message 没打穿。bg rendezvous 的公开消息类型里，reply 最终是 user/prompt 队列。 要硬插 assistant 需要拿到 live message array、React state setter、transcript mirror writer 或 agent core 的 history handle。 Bun inspector 目前缺 Runtime.queryObjects、function-call breakpoint、启动暂停，导致纯 inspector 抢闭包路线卡住。

Codex explicit app-server TUI 控制链路

Codex 这边比 Claude 更直接：开源代码里有 app-server JSON-RPC 控制面，TUI 自己也可以作为 app-server client。 这不是 codex exec 无头模式，而是有头 TUI 连接远端 app-server。 

codex app-server --listen ws://127.0.0.1:19325
codex --remote ws://127.0.0.1:19325 \
  --dangerously-bypass-approvals-and-sandbox \
  -C "$PWD" \
  --no-alt-screen

外部第三进程也连同一个 WebSocket，发 JSON-RPC。

实验：外部 turn/start + turn/steer

-> ["019ec071-280d-7a83-91ac-aa49561360be"]

  turnId = 019ec072-3044-77a1-b617-5a39ed47a550

再用 thread/read 读结构化状态，最后一个 turn 内确实出现两个 user message：

这证明外部进程在 TUI 正在承载的 Codex turn 中途插入了 user input，且模型最终把两个 marker 都响应出来。

实验：raw assistant 注入

Codex app-server 还有更直接的 thread/inject_items：

thread/inject_items:
  assistant message = CODEX_REMOTE_RAW_ASSISTANT_180739

next turn asks model to repeat injected assistant token

AGENT_ITEM text="CODEX_REMOTE_RAW_ASSISTANT_180739"

这比 Claude bg rendezvous 更强：Codex 官方控制面支持向 thread 结构注入 raw assistant item， 后续模型上下文会看到这条被注入的 assistant message。

Codex 配置和默认 remote 问题

用户追问 --remote 等参数能否通过环境变量或配置固化，让普通 codex 不手动加参数也连到外部控制面。 结论分两层。

明确不能配置的部分

--remote 是纯 CLI option，TUI 启动收到的是已经解析好的 explicit_remote_endpoint。 当前没有官方 config.toml 或环境变量把任意 remote endpoint 变成默认。

隔离 CODEX_HOME 反证实验：

remote = "ws://127.0.0.1:19325"

[app_server]
listen = "ws://127.0.0.1:19325"

codex --strict-config exec ... 报未知配置字段：

unknown configuration field `remote`
unknown configuration field `app_server`

可配置的外围参数

# ~/.codex/config.toml
approval_policy = "never"
sandbox_mode = "danger-full-access"

[tui]
alternate_screen = "never"

这些能固化审批、沙箱、alternate screen 等体验参数，但不能固化最关键的 remote endpoint。

Codex implicit local daemon reuse

这条是后续严查出来的关键。plain codex 在没有显式 --remote 时，并不总是自己启动 embedded server。 如果默认 Unix socket 已存在且可连接，它会复用这个本地 app-server。外部客户端连同一个 socket，就能控制 plain TUI 的 loaded thread。

默认 socket

~/.codex/app-server-control/app-server-control.sock

实验启动：

# 先启动默认 socket app-server
codex app-server --listen unix://

# 再启动完全不带 --remote 的 TUI
codex

TUI 清掉 update modal 之后，外部 WebSocket-over-Unix-socket probe 连接默认 socket， thread/loaded/list 看到了 plain TUI thread： 

019ec084-f4a3-7fd3-96bd-32d66b3c6a3a

外部 turn/start 证据

clientUserMessageId = CODEX_IMPLICIT_DAEMON_TURN_182701

thread/read includeTurns 读回：

userMessage.clientId = CODEX_IMPLICIT_DAEMON_TURN_182701
agentMessage.text   = CODEX_IMPLICIT_DAEMON_TURN_182701
turn.status         = completed
durationMs          = 3560

这证明 plain TUI 被外部 socket 注入 user turn，真实 API 跑完，并且结果进入 TUI thread 历史。

源码条件和负例

自动复用探测很朴素：默认 socket 文件存在，并且 UnixStream 在短时间内 connect 成功，就选择 local daemon。 它不是 JSON-RPC health check，所以假 socket 也可能诱导 TUI 走这条路，然后后续协议失败。

成立条件：

  • Unix 平台。
  • 默认 socket 存在：$CODEX_HOME/app-server-control/app-server-control.sock。
  • 没有显式 --remote。
  • 没有 -c key=value。
  • 没有 --strict-config、loader override、ignore-user-config、profile-v2 等。
  • 没有 --dangerously-bypass-hook-trust。

负例实测：codex -c 'web_search="live"' 启动后没有接入默认 socket； server socket fd 数不变，外部 thread/loaded/list 没有多出它的 thread。

官方 daemon lifecycle 限制

本机 codex app-server daemon start 和 codex remote-control start 当时失败，原因是安装形态： 

managed standalone Codex install not found at
~/.codex/packages/standalone/current/codex

daemon 子命令要求 install.sh 管理的 standalone Codex。npm 版仍可手动跑 codex app-server --listen unix://，只是不能使用官方 daemon lifecycle。

AgentBridge push to Claude Code 链路

AgentBridge 的 Claude 侧不是利用 bg rendezvous，也不是找现有 Claude 进程打洞。 它使用 Claude Code 的 Channels Research Preview：Claude Code 启动时主动加载 AgentBridge plugin， plugin 里的 MCP server 作为 Claude 的 child process 通过 stdio 与宿主通讯。

启动与 plugin

plugin MCP manifest：

{
  "mcpServers": {
    "agentbridge": {
      "command": "bun",
      "args": ["run", "${CLAUDE_PLUGIN_ROOT}/server/bridge-server.js"],
      "env": {
        "AGENTBRIDGE_DAEMON_ENTRY": "./daemon.js"
      }
    }
  }
}

agentbridge claude 启动 Claude 时加入：

--dangerously-load-development-channels plugin:agentbridge@agentbridge

MCP server 声明 experimental channel capability：

capabilities: {
  experimental: { "claude/channel": {} },
  tools: {},
}

Codex to Claude push 链路

Codex TUI
  --remote ws://127.0.0.1:<agentbridge-proxy>
      |
AgentBridge Codex proxy
  intercept item/agentMessage/delta + item/completed
      |
AgentBridge daemon
  WS frame: { type: "codex_to_claude", message }
      |
Claude plugin foreground bridge
  MCP stdio server
      |
server.notification({
  method: "notifications/claude/channel",
  params: { content, meta }
})
      |
Claude Code channel message

真正 push 代码形状：

await this.server.notification({
  method: "notifications/claude/channel",
  params: {
    content: message.content,
    meta: {
      chat_id: this.sessionId,
      message_id: message.id,
      delivery_attempt_id: deliveryAttemptId,
      user: "Codex",
      user_id: "codex",
      ts,
      source_type: "codex",
    },
  },
});

最后一跳是 MCP server notification，不是 PTY、输入框、transcript 或 hook。

Codex 侧来源

AgentBridge 自己启动 codex app-server --listen <listen>， 再让 Codex TUI 连自己的 proxy： 

--enable tui_app_server --remote <proxyUrl>

proxy 监听 app-server 通知：item/started、item/agentMessage/delta、item/completed。 当 item.type === "agentMessage" 完成时 emit： 

{
  id: item.id,
  source: "codex",
  content,
  timestamp: Date.now()
}

daemon 再把它转成 control WS 消息： 

{ type: "codex_to_claude", message }

Claude to Codex 反向路径

Claude plugin 暴露 MCP tools：reply、get_messages、get_budget。

Claude reply tool
  -> daemonClient.sendReply(...)
  -> WS { type: "claude_to_codex", ... }
  -> daemon
  -> codex.injectMessage(...)
  -> app-server JSON-RPC turn/start 或 turn/steer

对比我们的 Claude bg rendezvous 链路

AgentBridge 很干净、产品化，但 Claude 侧控制权来自“Claude 主动加载 plugin/channel”。 它不能控制一个未加载 AgentBridge channel 的普通 Claude TUI。

bg rendezvous 更接近底层外部控制：TUI 背后本来就有 daemon/background rendezvous primitive， 外部进程只要有 socket 就能贴进内部队列。Codex implicit daemon reuse 则更进一步： 普通 codex 可以在无显式参数时复用预先存在的 app-server control socket。

失败实验和排除项

  • claude --inspect 没打开 Bun inspector；有效入口是 BUN_INSPECT。
  • BUN_INSPECT=端口 不指定 path 时 WebSocket path 随机或不便直接连接；PORT/inspect 更稳。
  • expect PTY 下主动触发 /status 时 CDP backend 曾卡在 Runtime.evaluate；该结果被丢弃。后续用 Python pty harness 验证 CDP 正常。
  • Bun inspector 缺少 Runtime.queryObjects、setBreakpointOnFunctionCall 和可靠启动暂停，导致直接抢 bundle closure 变量没有完成。
  • 只设置 CLAUDE_BG_RENDEZVOUS_SOCK 不够；必须同时设置 CLAUDE_BG_BACKEND=daemon。
  • Claude assistant message 直接插入未打穿；当前可证实的是 user/prompt 队列注入、slash command、bash mode。
  • Codex TUI stdout 不是 terminal 时会拒绝启动；实验中改用 tmux 真 PTY，证据仍从 app-server 取。
  • plain codex 可能先被 update modal 阻塞；清掉 modal 后才能进入正常 TUI loop 并加载 thread。
  • codex app-server daemon start 在 npm 安装形态下失败，需要 standalone managed install；手动 app-server --listen unix:// 仍可验证协议。

安全和操作建议

  • 所有 socket path、debug port、heap dump 都应按本机高权限调试入口处理，避免放到共享目录或 repo。
  • 对 Claude rendezvous socket，启动脚本应使用私有目录，例如 mktemp -d 下的 socket，而不是固定公共路径。
  • 对 Codex app-server，默认 Unix socket 一旦存在，plain codex 可能自动复用；这对本机自动化很强，对误接入也有风险。
  • 保存 proof 时只保存 marker、状态摘要、结构化字段，不保存完整 thread/read、env、heap snapshot、debug log。
  • 如果要做长期工具，应把 transport、source of truth、wake/event contract 分清，不要把 WebSocket/Unix socket 当成架构本身。

后续最值得打的方向

  1. Claude assistant message 注入。 需要越过 bg rendezvous 的 user-only reply 语义，找到 live message store 或 agent history primitive。 可能路线是启动期源码改写、BunFS bundle patch、或更强的 JSC/Bun debug hook。
  2. Claude /status 完整结构化重建。 已拿到 model、effort、mcp、cwd、version 等核心项；还可继续把 auth/provider、permission mode、session name、diagnostics promise 分层还原。
  3. Codex implicit daemon 自动化。 写一个最小 control socket supervisor：先起 app-server unix socket，再启动 plain codex，外部进程订阅 thread/loaded/list 和 turn lifecycle。
  4. AgentBridge channel 落点验证。 需要确认 notifications/claude/channel 在 Claude Code 内部是 UI channel、model context channel，还是能触发 agent loop。 目前源码只证明它是 sanctioned push channel，不证明等价于 user turn 注入。
  5. 敏感 artifact 管理。 为 heap/debug/proof 建 out-of-repo 临时目录和自动 scrubber，避免把 token、prompt、session dump 混进项目。

最终判断

这次探索证明，Claude Code 和 Codex 都有不经过输入框的真实控制面，但性质不同。

Claude Code 的强路径是 background daemon rendezvous：它原本用于后台 agent 管理，手动挂到可见 TUI 后， 就变成 user prompt、slash command、bash mode 的本地 socket 注入通道。它官方意味很强，但不是普通前台默认暴露。

Codex 的强路径是 app-server：这是明确的 JSON-RPC 控制平面。显式 --remote 已经足够强； 默认 Unix socket implicit reuse 则更接近“普通 TUI 被外部控制”的形态。

AgentBridge 是一条产品化桥接：Codex 侧借 app-server，Claude 侧借 Channels preview。它干净、可包装， 但 Claude 侧需要 plugin/channel 预加载，所以底层控制力不如 bg rendezvous 或 Codex implicit daemon reuse。
